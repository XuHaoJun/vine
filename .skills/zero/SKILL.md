# Zero data layer (vine)

Vine syncs data with `@rocicorp/zero` 0.26 plus the `on-zero` 0.2 helpers. The canonical schema lives in `packages/zero-schema` (`@vine/zero-schema`); the web client wires it up in `apps/web/src/zero/client.tsx`; the Fastify server hosts pull/push at `apps/server/src/plugins/zero.ts`.

`apps/web/src/data/` is **dead legacy** — ignore it. Always work in `packages/zero-schema/src/`.

## When to reach for Zero

| Use Zero | Use something else |
|---|---|
| Anything backed by a table in `packages/zero-schema/src/models/` (chats, messages, friendships, todos, user profile, user state) | One-off REST/RPC calls, third-party APIs, file uploads, payment flows |
| Real-time, multi-client, or offline-friendly data | Stateless queries that don't need cache or sync |
| List → detail navigation that should feel instant | Long-running server actions with no client-visible row |

The data-fetching skill (`vine-data-fetching`) covers the broader Zero / TanStack Query / ConnectRPC tradeoff — load it if you're choosing between layers.

## Project layout

```
packages/zero-schema/src/
├── models/          one file per table — schema + permissions + mutate
│   ├── chat.ts, chatMember.ts, friendship.ts, message.ts,
│   ├── todo.ts, user.ts (table name: userPublic), userState.ts
├── queries/         zql query functions, one file per table
├── relationships.ts every relationship in one place, exported as allRelationships
├── schema.ts        createSchema({ tables, relationships, enableLegacyQueries: false })
├── server/          server-only actions + zero migrations
├── generated/       AUTO-GENERATED by `bun zero:generate` — never hand-edit
│   ├── tables.ts, models.ts, types.ts,
│   ├── syncedQueries.ts, groupedQueries.ts, syncedMutations.ts
├── augment.ts       declares Schema/AuthData/ServerActions onto on-zero's Config
└── types.ts         AuthData + hand-written *WithRelations helper types
```

The web app imports from the package and from its local client wrapper:

```ts
import { todosByUserId } from '@vine/zero-schema/queries/todo'
import { useZeroQuery, zero } from '~/zero/client'
```

`~/zero/client` re-exports from `createZeroClient(...)`:
- `useZeroQuery` — React hook for queries
- `zero` — mutation client (`zero.mutate.<table>.<action>(...)`)
- `getQuery` — imperative query (rare; for non-React contexts)
- `usePermission` — check permissions inside a component
- `ProvideZero` — provider component (mounted near the root)
- `zeroEvents` — error/warning event bus
- `run` — re-export from `on-zero` for one-off awaited reads

## Models

A model file contains three exports: the table schema, a permission, and a mutate object. Look at `packages/zero-schema/src/models/todo.ts` for the simplest shape:

```ts
import { boolean, number, string, table } from '@rocicorp/zero'
import { mutations, serverWhere } from 'on-zero'
import type { TableInsertRow } from 'on-zero'

export type Todo = TableInsertRow<typeof schema>

export const schema = table('todo')
  .columns({
    id: string(),
    userId: string(),
    text: string(),
    completed: boolean(),
    createdAt: number(),
  })
  .primaryKey('id')

const permissions = serverWhere('todo', (_, auth) => {
  return _.cmp('userId', auth?.id || '')
})

export const mutate = mutations(schema, permissions)
```

Passing `(schema, permissions)` to `mutations()` auto-generates CRUD: `zero.mutate.todo.insert(row)`, `.update({ id, ...patch })`, `.delete({ id })`. The permission gates both reads (via the synced query) and writes.

**Column types** (from `@rocicorp/zero`): `string()`, `number()`, `boolean()`, `json()`. Append `.optional()` for nullable. Booleans in vine are sometimes stored as `number()` 0/1 when the column also feeds Postgres types that need integers — see `chat.requireApproval`. Match the existing column type from `@vine/db` rather than inventing one.

**Naming gotcha:** the user table is registered as `table('userPublic')`, so client-side queries use `zql.userPublic` and mutations use `zero.mutate.userPublic.*`. The model file is `models/user.ts` only because there's also a private auth table elsewhere.

After editing or adding a model, run codegen:

```bash
bun --filter @vine/zero-schema zero:generate
# or, if you're already inside packages/zero-schema:
bun zero:generate
```

This rewrites `src/generated/`. Never edit those files; they're rebuilt every run.

### Custom mutations

Pass a third argument to `mutations()` to override or add named actions. The same context (`{ authData, can, tx }`) is available, plus `tx.query` for reading other tables and `tx.mutate` for writing them. The pattern from `models/message.ts`:

```ts
export const mutate = mutations(schema, messageReadPermission, {
  send: async ({ authData, tx }, message: Message) => {
    if (!authData) throw new Error('Unauthorized')

    if (message.senderType === 'user' && message.senderId !== authData.id) {
      throw new Error('Unauthorized')
    }

    await tx.mutate.message.insert(message)
    await tx.mutate.chat.update({
      id: message.chatId,
      lastMessageId: message.id,
      lastMessageAt: message.createdAt,
    })
  },
})
```

Two non-obvious things vine code does inside custom mutations:

1. **Cross-table reads use `tx.query` cast to `Record<string, any>`.** The narrow type on `tx.query` doesn't include other tables yet, so vine code does:
   ```ts
   const query = tx.query as Record<string, any> | undefined
   if (query?.chatMember) {
     const rows = await query.chatMember.where('chatId', chatId).run()
   }
   ```
   See `models/chat.ts` (`findOrCreateDirectChat`, `createGroupChat`) and `models/chatMember.ts` for canonical examples. The `if (query?.chatMember)` guard exists because `tx.query` is undefined on the optimistic client path — the dedup/role checks then degrade gracefully on the client and run for real on the server.

2. **`can(permission, key)` for explicit authz** — used in `models/user.ts` and `models/friendship.ts` when the default mutation permission isn't enough. Throw `Error('Unauthorized')` whenever auth fails; the message bubbles up to the caller.

Custom mutations are how vine implements all the chat/group flows: `chat.findOrCreateDirectChat`, `chat.createGroupChat`, `chatMember.markRead`, `chatMember.joinViaInvite`, `friendship.accept` (which also creates the chat). Reach for them whenever a single user-facing action needs to touch multiple tables atomically.

## Queries

Each table has a query file at `packages/zero-schema/src/queries/{table}.ts`. Queries are plain functions taking a `props` object and returning a zql builder:

```ts
import { zql } from 'on-zero'
import { messageReadPermission } from '../models/message'

export const messagesByChatId = (props: { chatId: string; limit?: number }) => {
  return zql.message
    .where(messageReadPermission)
    .where('chatId', props.chatId)
    .related('sender')
    .orderBy('createdAt', 'asc')
    .limit(props.limit ?? 100)
}
```

Patterns to copy from existing queries:
- Always start with `.where(<permission>)` so the server filters before sending data over the wire. Either import the model's exported permission (e.g. `chatReadPermission`) or define a local one with `serverWhere` in the query file (todo and user do this).
- Use `.related('relName', q => ...)` to load joined data in one round-trip; nest `.related()` inside the callback for multi-hop loads.
- Use `.one()` to get a single row instead of an array; combine with a primary-key where for detail queries.
- Use `.orderBy(...)` and `.limit(...)` to keep the synced set small.
- For OR / complex predicates, pass a callback to `where`: `.where(eb => eb.or(eb.cmp('a', x), eb.cmp('b', x)))` — see `queries/friendship.ts`.

After adding a query function, regenerate so it shows up in `generated/syncedQueries.ts` (the server validators) and `generated/groupedQueries.ts` (the client registry):

```bash
bun --filter @vine/zero-schema zero:generate
```

### `useZeroQuery` in React

```tsx
import { useZeroQuery } from '~/zero/client'
import { messagesByChatId } from '@vine/zero-schema/queries/message'

const [messages, { type }] = useZeroQuery(
  messagesByChatId,
  { chatId },
  { enabled: Boolean(chatId) }, // skip the query until the input is ready
)

const isLoading = type === 'unknown'
```

`type` is `'unknown'` while the local cache is still warming, `'complete'` once the server has confirmed the result set. Show a loading state on `'unknown'`. Almost every hook in `apps/web/src/features/**/use*.ts` follows this pattern; copy from `useTodos`, `useChats`, `useMessages`, or `useFriendship` rather than inventing a new style.

## Permissions (`serverWhere`)

`serverWhere` is the only place auth logic lives — it runs on the server, never on the client. The first argument is a query builder; vine code names it `_` for simple comparisons and `eb` ("expression builder") when relationships or boolean composition are involved. Both names work; match the surrounding file.

```ts
// owns the row
serverWhere('todo', (_, auth) => _.cmp('userId', auth?.id || ''))

// admin OR self
serverWhere('userPublic', (_, auth) =>
  _.or(_.cmpLit(auth?.role || '', '=', 'admin'), _.cmp('id', auth?.id || ''))
)

// public read
serverWhere('userPublic', () => true)

// relationship-gated read: only members of a chat can see it
serverWhere('chat', (eb, auth) =>
  eb.exists('members', q => q.where('userId', auth?.id || ''))
)
```

A few things to internalise:

- The `auth?.id || ''` fallback is intentional — when there's no logged-in user, the comparison returns no rows instead of throwing. Don't replace with `auth!.id`.
- `eb.exists('relationName', q => ...)` requires the relationship to be defined in `relationships.ts` first. The chat / message permissions both lean on this; that's why `chat` and `message` declare a `members` relationship even though product code rarely traverses it.
- Permissions exported from the model (e.g. `chatReadPermission`, `messageReadPermission`, `friendshipPermission`) are reused in the query files. Keep them named and exported instead of redefining the same predicate.

## Relationships

All relationships live in one file: `packages/zero-schema/src/relationships.ts`. Import from `generated/tables` (not from `models/`) — that's how Zero gets the right schema reference without circular types.

```ts
import { relationships } from '@rocicorp/zero'
import * as tables from './generated/tables'

export const messageRelationships = relationships(tables.message, ({ many, one }) => ({
  sender: one({
    sourceField: ['senderId'],
    destSchema: tables.userPublic,
    destField: ['id'],
  }),
  members: many({
    sourceField: ['chatId'],
    destSchema: tables.chatMember,
    destField: ['chatId'],
  }),
}))

export const allRelationships = [
  userRelationships,
  todoRelationships,
  // ...
  messageRelationships,
]
```

Whenever you add a new `relationships(...)` call, append it to `allRelationships`. `schema.ts` consumes that array, so forgetting it means your `.related()` and `eb.exists()` calls compile but return nothing.

Once defined, use it from queries (`.related('sender')`) or permissions (`eb.exists('members', q => ...)`).

## Mutations from React

Use the `zero.mutate.<table>.<action>(...)` API. For tables that only use the auto-generated CRUD, the actions are `insert`, `update`, `delete`. For tables with custom `mutations({...})`, the custom names are added. Examples:

```ts
zero.mutate.todo.insert({ id, userId, text, completed: false, createdAt })
zero.mutate.todo.update({ id, completed: true })
zero.mutate.todo.delete({ id })

zero.mutate.message.send({ id, chatId, senderId, senderType: 'user', type: 'text', text, createdAt })
zero.mutate.chatMember.markRead({ id: membershipId, lastReadMessageId, lastReadAt })

await zero.mutate.chat.createGroupChat({ chatId, name, memberIds, requireApproval, createdAt })
```

Calls without `await` are optimistic: the local cache updates immediately and the server confirms in the background. `await zero.mutate.x.y(args)` waits for the server round-trip — use it when the next step needs the server's "yes" (e.g. closing a dialog, navigating, showing an error toast). Vine reaches for `await` in the dialog flows (`CreateGroupDialog`, `InviteLinkDialog`); fire-and-forget elsewhere.

### Convergence: caller generates IDs and timestamps

Every mutation runs **twice**: once on the client (optimistic) and once on the server. They must produce byte-identical writes — if the function generates a fresh `id` or `Date.now()` on each run, the optimistic row diverges from the canonical one and sync gets stuck.

The vine convention: the React caller generates `crypto.randomUUID()` and `Date.now()` and passes them in. The mutation just stores what it was given.

```ts
// in a hook (apps/web/src/features/todo/useTodos.ts)
const addTodo = (text: string) => {
  zero.mutate.todo.insert({
    id: crypto.randomUUID(),
    userId,
    text,
    completed: false,
    createdAt: Date.now(),
  })
}
```

Custom mutations that need to create multiple rows accept all the IDs/timestamps as args. See `friendship.accept`, `chat.findOrCreateDirectChat`, and `chat.createGroupChat` — every chat/member id is generated by the caller (`crypto.randomUUID()`) and passed in.

`Date.now()` inside the *body* of a custom mutation is acceptable when the value is only used for derived/server-only state (e.g. `now` in `friendship.accept` is fine because the chat row's `createdAt` is also generated server-side and the chat is brand new). When in doubt, accept the timestamp from the caller to keep things deterministic.

### Server-only side effects

Side effects that shouldn't run on the optimistic client path live behind `if (ctx.server)`. The current vine codebase doesn't actually use this pattern in any model yet; if you need an analytics ping or push notification fire-off, the supported shape is:

```ts
if (ctx.server) {
  ctx.server.asyncTasks.push(async () => {
    await ctx.server.actions.analyticsActions().logEvent(ctx.authData.id, 'todo_created')
  })
}
```

`asyncTasks` runs after the transaction commits. Don't put cache-affecting writes here — they'd silently skip on the client and cause divergence.

## Server-side wiring

You almost never edit these files but it helps to know they exist:

- `apps/server/src/plugins/zero.ts` mounts `/api/zero/pull` and `/api/zero/push`, calling `createZeroServer({ schema, models, createServerActions, queries, database })`.
- `packages/zero-schema/src/server/createServerActions.ts` registers `analyticsActions` and `userActions`. New cross-table server logic (calling Drizzle, etc.) goes in `packages/zero-schema/src/server/actions/<name>.ts` and is added to that registry.
- `packages/zero-schema/src/augment.ts` declares the `Schema`, `AuthData`, and `ServerActions` types onto `on-zero`'s `Config` interface so `tx.query`, `authData`, and `ctx.server.actions` are all typed.

## Anti-patterns to avoid

**Waterfall via `useUser()`** — `useUser()` waits on a DB round-trip; `useAuth()` reads from the JWT and is available immediately. Always derive `userId` from `useAuth()`:

```tsx
const { user } = useAuth()
const userId = user?.id ?? ''
const [todos] = useZeroQuery(todosByUserId, { userId }, { enabled: Boolean(userId) })
```

**N+1 queries inside list items** — load relations in the parent query with `.related()`, don't fire a per-row `useZeroQuery`.

**Client-side filtering of synced sets** — express it in the query with `where`, `eb.or`, or `eb.exists`. Two `useZeroQuery` calls just to subtract one from the other is both slow and a permission leak risk.

**Editing files in `apps/web/src/data/`** — that directory is dead. The real schema is `@vine/zero-schema`. If you find yourself reading or writing `~/data/...`, stop and switch to `@vine/zero-schema/...`.

**Hand-editing `packages/zero-schema/src/generated/`** — gets blown away by `bun zero:generate`. If something needs to change in `generated/`, change the underlying model/query/relationship and re-run codegen.

**List queries that don't include detail data** — Zero's killer feature is instant list-to-detail navigation because the cache already has the row. Make the list query load the same `.related()` shape the detail page needs:

```ts
export const chatsByUserId = (props: { userId: string }) =>
  zql.chat
    .where(chatReadPermission)
    .related('members', q => q.related('user'))
    .related('lastMessage')
    .orderBy('lastMessageAt', 'desc')
    .limit(50)
```

## Workflow: adding a new table

1. Create the Drizzle migration in `packages/db/` and apply it (see the `zero-schema-migration` skill for the publication-rebuild dance — that part *is* required for new columns/tables).
2. Create `packages/zero-schema/src/models/<name>.ts` — `schema`, permission via `serverWhere`, and `export const mutate = mutations(schema, permission)`. Add a `TableInsertRow` type export.
3. Re-export the model from `packages/zero-schema/src/index.ts` (e.g. `export * as fooModel from './models/foo'`).
4. Create `packages/zero-schema/src/queries/<name>.ts` with the query functions you need.
5. Add relationships in `relationships.ts` and append them to `allRelationships`.
6. Run codegen: `bun --filter @vine/zero-schema zero:generate`.
7. Build the React hook in `apps/web/src/features/<domain>/use<Name>.ts` using `useZeroQuery` and `zero.mutate.<name>.*`. Match the patterns in `useTodos.ts` / `useChats.ts`.

## Workflow: adding a column to an existing table

This is the schema-migration flow; load the `zero-schema-migration` skill — it covers Drizzle migration, publication rebuild, and Zero server restart in detail. The model edit itself is just adding the column to the `.columns({...})` block and re-running `bun zero:generate`.

## Debugging

- `?debug=2` on the URL turns on verbose Zero sync logs in the browser console.
- `window.zero`, `window.run`, and `window.dropAllDatabases` are exposed in dev for poking at the client from DevTools.
- The "Reset Zero data" button surfaces via `clearClientData()` in `apps/web/src/zero/helpers/clearClientData.ts` — it deletes all IndexedDB databases that look like Zero/Replicache and reloads. Use when the local cache is stuck in a bad state after a schema change.
- Sync errors trigger `onClientStateNotFound` / `onUpdateNeeded`, which call `showClientDataErrorOnce` to notify the user and (within a 3-minute window) auto-clear and reload.
