## 1. 系統概述

LINE POINTS 是一套與 LINE 生態系整合的**虛擬點數獎勵系統**，用於：

- 回饋用戶消費行為（LINE Pay、合作通路）
- 激勵用戶完成指定任務（廣告任務牆）
- 提升用戶留存（可折抵消費，增加回購誘因）
- 作為行銷工具（企業可購買點數序號發送給消費者）

**核心概念：** 點數與帳號深度綁定，不可轉讓，可折抵消費或兌換商品/服務。

---

## 2. 點數類型

LINE POINTS 分為兩種類型，需分開儲存與管理：

| 類型 | 說明 | 有效期限規則 |
|------|------|-------------|
| **一般點數** | 透過 LINE 各服務活動、任務牆獲得 | 最後一次獲得起算 180 天，可滾動延長 |
| **限時點數** | 由特定行銷活動（如特定信用卡消費）產生 | 每筆獨立計算，自獲得日起 180 天，**不可延長** |

> **設計提示：** 資料庫中建議以 `point_type` 欄位區分，並為限時點數的每一筆記錄儲存獨立到期日。

---

## 3. 點數獲得途徑

### 3.1 消費回饋
- 使用 LINE Pay 於合作通路消費，依比例回饋點數
- 合作通路涵蓋：購物平台、旅遊、外送、交通等

### 3.2 任務牆（Task Wall）
- 用戶完成指定任務即可獲得點數
- 常見任務類型：
  - 加入官方帳號好友（1～5 點）
  - 觀看指定廣告影片（1～5 點）
  - 下載指定 App 並達成目標（1～20 點）
  - 填寫問卷 / 申請試用品
  - 首次消費 / 新會員登錄

### 3.3 序號兌換（Point Code）
- 企業購買點數序號，作為行銷活動獎品
- 用戶至指定頁面輸入英數混合序號兌換
- 序號為一次性使用，兌換後立即入帳

### 3.4 其他來源
- 退款退還（交易取消後）
- 特定活動抽獎 / 遊戲獎勵

## 4. 點數使用方式

### 4.1 消費折抵（主要使用場景）
- **比例：** 1 點 = 1 元（台幣），無上限
- **前提：** 用戶須於設定頁面手動開啟「點數折抵」功能
- **適用範圍：** 所有支援 LINE Pay 的實體 / 線上通路

### 4.2 購買虛擬商品（自動轉換）
- 購買貼圖、主題、表情貼時，若代幣不足，系統**自動**將點數換算為代幣扣除
- 台灣匯率：**2 枚代幣 = 1 點**（無法手動預先兌換）

### 4.3 兌換商品 / 優惠券
- 至 LINE POINTS 兌換頁面換取：
  - 超商咖啡券、即享券
  - 品牌折價券
  - 慈善捐款（以點數折算捐款金額）

### 4.4 第三方點數轉換
- 可將 LINE POINTS 轉換為合作平台點數（如：亞洲萬里通、Happy Go、OpenPoints 等）

## 5. 點數有效期限規則

### 5.1 一般點數
```
有效期限 = 最後一次「獲得」點數日期 + 180 天
```

- 只要**任何一筆**新點數入帳，整批一般點數的效期就會重新起算 180 天
- 即使只獲得 1 點，也能延長所有一般點數效期
- 效期到期後，系統**立即回收**，不另行保留緩衝期

### 5.2 限時點數
```
有效期限 = 獲得日期 + 180 天（每筆獨立計算，不可延長）
```

- 每筆限時點數的到期日各自獨立
- 不受新點數入帳影響

### 5.3 到期處理
- 超過有效期限的點數由系統自動回收（無法恢復）
- 兌換券逾期未使用，**不退還**點數

---

## 6. 點數折抵順序

採用 **先進先出（FIFO）** 原則：

> 有效期限相同的情況下，優先使用**較早獲得**的點數

**建議實作邏輯：**
1. 先消耗最接近到期的點數批次
2. 同到期日的點數依入帳時間排序，先入先扣
3. 限時點數與一般點數分開計算，按各自的到期順序消耗

## 7. 點數兌換與轉換

### 7.1 點數 → 代幣（虛擬貨幣）
| 地區 | 兌換率 |
|------|--------|
| 台灣 | 1 點 = 2 枚代幣 |
| 泰國 | 1 點 = 2 枚代幣 |
| 日本 | 3 點 = 1 枚代幣 |
| 韓國 | 20 點 = 1 枚代幣 |
| 印尼 | 300 點 = 1 枚代幣 |
| 其他 | 2 點 = 1 枚代幣 |

- 僅在購買時點數不足代幣的情況下**自動觸發**，不支援手動預先兌換

### 7.2 點數 → 第三方點數
- 需透過官方合作的點數交換頁面操作
- 轉換比例依各合作平台協議而定
- 轉換後**不可反轉**

## 8. 退款與沖銷機制

### 8.1 交易取消退款
- 退款點數處理時間：交易取消後約 **5～7 個工作天**
- 退還的點數會**立即延長整批點數效期**（自退款日起 +180 天）

### 8.2 點數餘額為負值的情境
- 若用戶在退款完成前已花光點數，退款點數入帳時帳戶會顯示**負值**
- 負值狀態不影響正常使用，後續入點自動補平

### 8.3 兌換商品退款
- 若已兌換優惠券逾期未使用，**不退還**點數
- 正常取消訂單（兌換券未使用），依通路規定處理退點

---

## 9. 通知機制

| 觸發條件 | 通知內容 | 通知管道 |
|----------|----------|----------|
| 點數於 **30 天內**到期 | 點數即將到期警告 + 當前餘額 | 官方帳號訊息推播 |
| 點數於 **7 天內**到期 | 緊急到期警告 | 官方帳號訊息推播 |
| 點數入帳 | 入帳確認通知 + 可查看詳情 | 官方帳號訊息推播 |
| 退款成功 | 退款點數到帳通知 | 官方帳號訊息推播 |

> **注意：** 通知訊息內標示的「即將到期點數」為訊息發送當下的快照，若想取得最新資訊需點入查看。

---

## 10. 帳號與點數綁定

- 點數**嚴格綁定**用戶帳號，無法轉移給他人
- 換手機 / 換裝置：只要完成**帳號移轉**操作，點數隨帳號保留
- 帳號移轉失敗或帳號喪失存取權，帳號內點數**無法取回**
- 不支援合併多個帳號的點數

---

## 11. 交易記錄

用戶可查看完整的點數流水帳，應包含：

| 欄位 | 說明 |
|------|------|
| 交易時間 | 點數入帳 / 扣除的時間戳記 |
| 交易類型 | 獲得 / 使用 / 到期回收 / 退款 |
| 點數變動量 | 正值（獲得）或負值（使用） |
| 來源 / 用途 | 獲得管道或消費項目 |
| 當下餘額 | 交易後的剩餘點數 |
| 到期日 | 該批次點數的有效期限 |

## 12. 資料模型建議

### 用戶點數帳戶（`point_accounts`）

```sql
CREATE TABLE point_accounts (
  id            BIGINT PRIMARY KEY,
  user_id       BIGINT NOT NULL UNIQUE,
  total_balance INT NOT NULL DEFAULT 0,         -- 一般點數餘額
  temp_balance  INT NOT NULL DEFAULT 0,         -- 限時點數餘額（總和，細節見 temp_points）
  last_earned_at TIMESTAMP,                     -- 最後一次獲得一般點數時間
  general_expires_at TIMESTAMP,                 -- 一般點數到期日（= last_earned_at + 180d）
  created_at    TIMESTAMP NOT NULL,
  updated_at    TIMESTAMP NOT NULL
);
```

### 點數交易記錄（`point_transactions`）

```sql
CREATE TABLE point_transactions (
  id            BIGINT PRIMARY KEY,
  user_id       BIGINT NOT NULL,
  amount        INT NOT NULL,                   -- 正值獲得，負值扣除
  point_type    ENUM('general', 'temporary'),
  transaction_type ENUM('earn', 'spend', 'expire', 'refund', 'convert'),
  source        VARCHAR(100),                   -- 來源描述（任務名稱、通路名稱等）
  reference_id  VARCHAR(100),                   -- 外部關聯 ID（訂單 ID、序號等）
  expires_at    TIMESTAMP,                      -- 此筆點數的到期日（限時點數用）
  balance_after INT NOT NULL,                   -- 交易後餘額快照
  created_at    TIMESTAMP NOT NULL
);
```

### 限時點數批次（`temporary_point_batches`）

```sql
CREATE TABLE temporary_point_batches (
  id         BIGINT PRIMARY KEY,
  user_id    BIGINT NOT NULL,
  amount     INT NOT NULL,                      -- 剩餘可用量
  expires_at TIMESTAMP NOT NULL,                -- 此批次到期日
  source     VARCHAR(100),
  created_at TIMESTAMP NOT NULL
);
```

### 點數序號（`point_codes`）

```sql
CREATE TABLE point_codes (
  id            BIGINT PRIMARY KEY,
  code          VARCHAR(50) NOT NULL UNIQUE,
  point_amount  INT NOT NULL,
  point_type    ENUM('general', 'temporary'),
  campaign_id   BIGINT,                         -- 所屬行銷活動
  valid_from    TIMESTAMP NOT NULL,
  valid_until   TIMESTAMP NOT NULL,
  redeemed_by   BIGINT,                         -- 兌換用戶 ID
  redeemed_at   TIMESTAMP,
  created_at    TIMESTAMP NOT NULL
);
```

## 13. API 端點建議

### 查詢餘額
```
GET /api/v1/points/balance
Response: { general: 350, temporary: 120, general_expires_at: "2025-10-01" }
```

### 查詢交易記錄
```
GET /api/v1/points/transactions?page=1&limit=20&type=earn
```

### 獲得點數（內部觸發）
```
POST /api/v1/points/earn
Body: { user_id, amount, point_type, source, reference_id }
```

### 使用點數
```
POST /api/v1/points/spend
Body: { user_id, amount, order_id, description }
```

### 兌換序號
```
POST /api/v1/points/redeem-code
Body: { code }
```

### 退款點數
```
POST /api/v1/points/refund
Body: { transaction_id, amount }
```

### 查詢即將到期點數
```
GET /api/v1/points/expiring?days=30
Response: { expiring_amount: 50, expires_at: "2025-05-10" }
```

## 14. 業務限制與邊界條件

| 項目 | 規則 |
|------|------|
| 點數轉移 | 不可轉讓給其他用戶 |
| 點數最小單位 | 整數（1 點），不支援小數 |
| 負值餘額 | 退款沖銷時允許暫時為負值，不阻擋後續入點 |
| 序號使用 | 每個序號僅限兌換一次 |
| 兌換券逾期 | 逾期不退點 |
| 最低使用門檻 | 折抵消費無最低限制（依業務決定） |
| 點數上限 | LINE 未明確限制，建議系統設定防呆上限（如 9,999,999 點） |
| 並發扣點 | 需使用樂觀鎖或資料庫事務防止超扣 |
| 到期批次任務 | 建議每日凌晨執行排程，回收到期點數並發送通知 |

## 附錄：LINE POINTS 與 LINE STORE 儲值金比較

| 項目 | LINE POINTS | LINE STORE 儲值金 |
|------|-------------|-----------------|
| 獲得方式 | 消費回饋、任務、序號 | 購買實體點數卡 / PIN 碼 |
| 使用場景 | LINE Pay 折抵、兌換商品 | 網頁版 LINE STORE 購買貼圖 |
| 有效期限 | 180 天滾動 | 依儲值卡規定 |
| 可轉讓 | 否 | 否 |
| 幣值 | 1 點 ≒ 1 元 | 依購買面額 |
